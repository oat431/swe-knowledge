# 02 Professionalism & Customer Expectations

Agile is often adopted for speed and quality. Those results don't show up immediately and developers get disappointed. The more important reasons: **professionalism** and **meeting reasonable customer expectations.**

---

## Professionalism

> Disciplined, professional behavior becomes more important as software becomes more important.

Software is everywhere. Bad software can kill people. Programmers will be blamed — and the disciplines of Agile development are a first step toward professionalism that might save lives.

**Key distinction:** High commitment to discipline is more important than ceremony. Scrum ceremonies without TDD and refactoring are empty ritual.

---

## 10 Reasonable Customer Expectations

Customers have legitimate expectations. Agile aims to meet them all:

| Expectation | What It Means |
|------------|--------------|
| **Don't ship bad software** | Users shouldn't need to think like programmers. High quality for good money. |
| **Continuous technical readiness** | System must be deployable at end of every iteration. Code clean, tests pass. Deploying = business decision, not technical. |
| **Stable productivity** | Progress shouldn't slow as codebase grows. Avoid big redesigns — they usually fail. |
| **Inexpensive adaptability** | Software is supposed to be "soft" — easy to change. Changing requirements is why software engineering exists. |
| **Continuous improvement** | Design, architecture, code structure, efficiency should improve over time, not deteriorate. |
| **Fearless competence** | Developers shouldn't fear modifying code. TDD gives confidence — "you touch it, you don't break it." |
| **No QA findings** | Bugs shouldn't be found by QA — they should be prevented by development. QA finds bugs → fix the process. |
| **Test automation** | Manual tests are expensive, skipped under budget pressure, and immoral for repetitive work. Machines test better. |
| **Cover for each other** | Developers act as a team. If someone fails or gets sick, others cover. Share knowledge. |
| **Honest estimates** | Provide ranges ("5–15 days"), not exact numbers ("10 days"). Under uncertainty, honest uncertainty is professional. |

---

## Saying No & Continuous Learning

Two additional professional obligations:

- **Saying "No":** If no feasible solution exists, the developer must say so. Inconvenient now, but saves bigger trouble later.
- **Continuous learning:** The industry changes fast. Company training is great — but the responsibility for learning remains with the developer.
- **Mentoring:** Existing team members must teach new ones. Teaching is the best way to learn.
