**Priority Legend:**

| Icon | Level | Meaning |
|---|---|---|
| 🔴 | **Must Have** | Essential for virtually all initiatives; skipping creates significant risk |

## 1. BA Planning & Monitoring

| Document | Description | Priority | ISO/IEEE Reference |
|---|---|---|---|
| **Business Analysis Approach** | Defines the method, timing, deliverables, and stakeholders for BA activities | 🔴 Must Have | ISO/IEC/IEEE 12207, ISO 9001 |
| **Stakeholder Engagement Approach** | Stakeholder list, roles, communication needs, and collaboration plan | 🔴 Must Have | ISO 21502 |
| **Governance Approach** | Decision-making process: approvals, prioritization, change control, roles/authority | 🔴 Must Have | ISO 9001 |
| **Information Management Approach** | How BA information will be stored, accessed, versioned, and reused | 🔴 Must Have | ISO/IEC/IEEE 12207 |
| **BA Performance Assessment** | Assessment of BA work effectiveness; identifies improvements for future initiatives | 🟡 Nice to Have | ISO/IEC 33000 (SPICE) |

## 2. Strategy Analysis

| Document | Description | Priority | ISO/IEEE Reference |
|---|---|---|---|
| **Current State Description** | Enterprise scope, capabilities, resources, performance, culture, dependencies, external influences | 🔴 Must Have | ISO/IEC/IEEE 12207, ISO/IEC/IEEE 15288 |
| **Business Requirements** | Problems, opportunities, or constraints; the "why" behind the change | 🔴 Must Have | ISO/IEC/IEEE 29148 |
| **Business Objectives** | SMART goals and measurable outcomes the future state will achieve | 🔴 Must Have | ISO/IEC/IEEE 29148 |
| **Future State Description** | Boundaries of proposed new, removed, and modified enterprise components; expected potential value | 🔴 Must Have | ISO/IEC/IEEE 12207, ISO/IEC/IEEE 15288 |
| **Potential Value** | Net benefit of solution after operating costs; benchmark for evaluating options | 🔴 Must Have | ISO 21505 |
| **Risk Analysis Results** | Identified risks with likelihood, impact, and mitigation strategies | 🔴 Must Have | ISO 31000 |
| **Change Strategy** | High-level plan: context, alternatives, justification, investment, transition states, value realization | 🔴 Must Have | ISO/IEC/IEEE 12207 |
| **Solution Scope** | Boundaries of the solution — capabilities, technology, processes, data, resources to be delivered | 🔴 Must Have | ISO/IEC/IEEE 12207 |
| **Gap Analysis** | Difference between current and future state capabilities | 🟡 Nice to Have | ISO/IEC/IEEE 15288 |
| **Enterprise Readiness Assessment** | Cultural readiness, operational readiness, resource capacity | 🟡 Nice to Have | ISO 21502 |

## 3. Elicitation & Collaboration

| Document | Description | Priority | ISO/IEEE Reference |
|---|---|---|---|
| **Elicitation Activity Plan** | Logistics, scope, techniques, and supporting materials for each elicitation activity | 🔴 Must Have | ISO/IEC/IEEE 29148 |
| **Elicitation Results (unconfirmed)** | Captured information in format specific to the elicitation activity | 🔴 Must Have | ISO/IEC/IEEE 29148 |
| **Elicitation Results (confirmed)** | Integrated output that BA and stakeholders agree correctly reflects captured information | 🔴 Must Have | ISO/IEC/IEEE 29148 |
| **Business Analysis Information (communicated)** | Information packaged and delivered to stakeholders in formats they can understand and use | 🔴 Must Have | ISO/IEC/IEEE 29148 |
| **Stakeholder Engagement** | Willingness from stakeholders to engage in BA activities; ongoing relationship health | 🔴 Must Have | ISO 21502 |

## 4. Requirements Analysis & Design Definition

| Document | Description | Priority | ISO/IEEE Reference |
|---|---|---|---|
| **Requirements (specified & modelled)** | Requirements and designs in text, matrices, and diagrams (use cases, user stories, process models, data models, prototypes) | 🔴 Must Have | ISO/IEC/IEEE 29148, ISO/IEC 19501 (UML) |
| **Requirements (verified)** | Requirements/designs checked for quality characteristics: atomic, complete, consistent, feasible, unambiguous, testable | 🔴 Must Have | ISO/IEC/IEEE 29148, ISO/IEC 20246 |
| **Requirements (validated)** | Requirements/designs demonstrated to deliver stakeholder benefit and align with business objectives | 🔴 Must Have | ISO/IEC/IEEE 29148 |
| **Requirements Architecture** | Structure of all requirements and their interrelationships; viewpoints and views representing different stakeholder concerns | 🔴 Must Have | ISO/IEC/IEEE 42010 |
| **Solution Recommendation** | The suggested, most appropriate solution based on evaluation of all design options; maximizes enterprise value | 🔴 Must Have | ISO/IEC/IEEE 12207 |
| **Design Options** | Various ways to satisfy needs — solution approach, improvement opportunities, allocated components | 🟡 Nice to Have | ISO/IEC/IEEE 12207 |

## 5. Requirements Life Cycle Management

| Document | Description | Priority | ISO/IEEE Reference |
|---|---|---|---|
| **Requirements (traced)** | Requirements with defined relationships to other requirements, solution components, releases/phases/iterations | 🔴 Must Have | ISO/IEC/IEEE 29148 |
| **Requirements (maintained)** | Requirements kept current, correctly attributed, and available for long-term reuse | 🔴 Must Have | ISO/IEC/IEEE 29148 |
| **Requirements (prioritized)** | Requirements ranked by relative importance — benefit, penalty, cost, risk, dependency, time sensitivity | 🔴 Must Have | ISO/IEC/IEEE 29148 |
| **Requirements (approved)** | Requirements agreed to by stakeholders, ready for construction or further work | 🔴 Must Have | ISO/IEC/IEEE 29148 |
| **Requirements Change Assessment** | Recommendation to approve, modify, or deny a proposed change with impact analysis | 🔴 Must Have | ISO/IEC/IEEE 29148, IEEE 828 |
| **Designs (traced/maintained/prioritized/approved)** | Same life cycle management applied to design components | 🔴 Must Have | ISO/IEC/IEEE 29148 |

## 6. Solution Evaluation

| Document | Description | Priority | ISO/IEEE Reference |
|---|---|---|---|
| **Solution Performance Measures** | Quantitative and qualitative KPIs providing information on how well the solution is performing | 🔴 Must Have | ISO/IEC 25010 (SQuaRE) |
| **Solution Performance Analysis** | Analysis of measurements with recommendations to solve performance gaps | 🔴 Must Have | ISO/IEC 25010 (SQuaRE) |
| **Solution Limitation** | Description of current solution-internal limitations: constraints, defects, component dependencies | 🔴 Must Have | ISO/IEC/IEEE 20000-1 |
| **Enterprise Limitation** | Description of current solution-external limitations: culture, operations, stakeholder factors | 🔴 Must Have | ISO/IEC/IEEE 20000-1 |
| **Recommended Actions** | What should be done to improve solution value: organizational change, retirement, enhancement | 🔴 Must Have | ISO/IEC/IEEE 20000-1 |

## How BABOK Fits with SWEBOK

```
BABOK (Business Analysis)                  SWEBOK (Software Engineering)
─────────────────────────                  ────────────────────────────
Strategy Analysis          ──feeds──→      Requirements
  Current State Desc                         SRS, BRD, User Stories
  Business Requirements
  Future State Desc
  Change Strategy / Solution Scope

Requirements Analysis      ──feeds──→      Architecture + Design
  Specified Requirements                     SAD, HLD, LLD, API Specs
  Requirements Architecture
  Design Options

Requirements Life Cycle    ──governs──→    Construction + Testing
  Traced / Maintained                        Source Code, Test Cases
  Prioritized / Approved

Solution Evaluation        ──validates──→  Operations + Maintenance
  Performance Measures                       Deployment Plan, SLAs
  Solution / Enterprise Limitations          Incident Mgmt, MR/PR
  Recommended Actions
```

> **Note:** Priority ratings are default recommendations. The BABOK does not prescribe a sequential process — tasks may be performed in any order based on initiative needs. All BABOK outputs should be adapted to the chosen methodology (predictive, adaptive/Agile, or hybrid).
